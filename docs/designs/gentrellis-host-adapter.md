# CEO Plan: GenTrellis Host Adapter

## Vision

### 10x Check
Not just "smol-agent inside GenTrellis" but the open standard for governed agent execution. A formal host contract spec (like OCI Runtime Spec) with conformance tests, multi-runtime support (smol-agent, LangGraph, Strands), protection-level-aware tool governance, and replayable audit traces. The governance is structural — built into the host contract, not bolted on.

### Platonic Ideal
You write an agent (in any framework), declare its tool requirements, push it to a governed host. The host ensures the agent can only do what it's allowed to do, routes LLM calls through the right models at the right protection levels, and produces a complete audit trail. It feels like deploying a Docker container — you think about what your app does, not the runtime. Governance is invisible until it blocks something dangerous, and then it's clear, specific, and auditable.

## Critical Finding: Tool Execution Gap

The agent loop in Agent.run() (agent.js:1237) calls registry.execute() directly, bypassing host.toolProvider.execute(). The host contract defines toolProvider but the actual execution path doesn't use it. This means the host contract is decorative — no external host can actually govern tool execution.

Similarly, session operations (agent.js:46-49) import directly from sessions.js instead of using host.sessionStore. And event emission uses Node EventEmitter (this.emit) instead of flowing through host.eventSink.

These gaps must be fixed before the GenTrellisHost adapter can work. They are prerequisites, not expansions.

Note: `Agent extends AgentRuntime` (agent.js:294) and `this.host` is already available via `super({ host })` (agent.js:310). However, tool execution is NOT just one seam — `registry.execute()` is also called directly in `architect.js:147` and `sub_agent.js:180`. Session operations are hard-wired in `agent.js:1497/1509/1541`, `session_tools.js`, and `App.js`. The fix is broader than a single call-site.

Additionally, `runtimeContext` (with tieredRouter fields) exists in session-metadata.js and request-context.js, but Agent never passes it into createProvider. The X-Workflow-Id and X-Protection-Level headers are defined but never sent — another decorative seam.

## Codex Outside Voice Findings (accepted)

- **Transport simplification:** ACP stdio already has streaming and could handle approvals. HTTP callbacks on top of ACP is redundant. **Decision: ACP-only transport for GenTrellis integration.**
- **CLI flag rename:** `--host` already means model provider base URL. **Decision: use `--agent-host` instead.**
- **Retry safety:** Many tool calls are writes/commands. Blind retry duplicates side effects. **Decision: classify safe vs unsafe retries; only retry read-only operations automatically.**
- **runtimeContext gap:** Agent never passes runtimeContext into createProvider. **Decision: wire through in Phase 1.**
- **Global singletons:** ACP hard-limits to one session due to global contamination. Multi-tenancy is a real blocker for the dashboard story. **Deferred to TODOS.md — acceptable for single-agent demo.**

## Accepted Scope (added to this plan)

Phase 1 — Fix the Foundation (smol-agent):
- Route tool execution through host.toolProvider.execute() (agent.js, architect.js, sub_agent.js)
- Route session operations through host.sessionStore (agent.js, session_tools.js)
- Route events through host.eventSink (dual-emit)
- Wire runtimeContext from host through to createProvider (so X-Workflow-Id and X-Protection-Level headers are actually sent)
- Governed sub-agents (inherit parent host)
- Protection badge in terminal UI
- Host Contract Conformance Test Suite (LocalHost passes). Concrete test cases:
  1. Host can deny a tool call via toolProvider (returns error, agent sees denial)
  2. Session create/load/save round-trips correctly through host.sessionStore
  3. Auditable events (tool_call, tool_result, response) appear in eventSink
  4. Memory read/write round-trips through host.memoryStore
  5. Tool execution results from host.toolProvider match direct registry.execute results (LocalHost parity)

Phase 2 — GenTrellis Integration:
- GenTrellisHost adapter (implements host contract via ACP stdio — NOT HTTP callbacks). GenTrellis spawns smol-agent as subprocess, communicates via ACP protocol which already handles streaming and can be extended for approvals. Auth: API token via `GENTRELLIS_AGENT_TOKEN` env var (scoped per-run). Error handling: classify safe (read-only) vs unsafe (write/command) operations; only auto-retry safe operations. Agent halts on persistent failure with structured error event.
  - Done when: conformance test suite passes with GenTrellisHost, and a governed agent run completes under Controlled protection level
- `--agent-host gentrellis://` CLI flag (renamed from --host to avoid collision with provider base URL). Auth: reads `GENTRELLIS_API_KEY` from env or `~/.config/smol-agent/hosts.json`.
  - Done when: `smol-agent --agent-host gentrellis://localhost:8000/workflow/1 "hello"` runs a governed agent
- Trace export as Markdown (.trace.md)
  - Done when: a complete agent run produces a human-readable trace file with tool calls, governance decisions, and timestamps
- Agent run API in GenTrellis (FastAPI endpoints)
  - Done when: POST /api/agents/runs creates a run, events stream via SSE, approval gate works
- Dashboard extension (attach agent, live trace, approval gates)
  - Done when: operator can attach agent to workflow, launch it, see live trace, and approve/deny tool calls
- HallMonitor integration for agent LLM calls
  - Done when: agent responses show protection badges (Protected/Blocked) matching HallMonitor review
